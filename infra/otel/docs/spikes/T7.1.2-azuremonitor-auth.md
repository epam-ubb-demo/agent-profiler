# T7.1.2 — Azure Monitor Exporter Authentication Method

| Field     | Value                                              |
| --------- | -------------------------------------------------- |
| **Status**  | Accepted                                         |
| **Date**    | 2025-01-10                                       |
| **Feature** | F7.1 — Infrastructure Foundation                 |
| **Task**    | T7.1.2 — Azure Monitor Authentication Selection  |

## Context

The OTel Collector must export telemetry (traces, metrics, logs) to Azure
Monitor / Application Insights via the `azuremonitor` exporter. Two
authentication approaches were evaluated:

1. **Connection string** — the traditional Application Insights instrumentation
   key and ingestion endpoint, supplied as a single connection string.
2. **Managed Identity (MI)** — Azure AD token-based authentication using the
   container's system- or user-assigned managed identity.

This spike determines which authentication method is production-ready for our
deployment.

## Investigation

### Connection string via environment variable

- The `azuremonitor` exporter natively supports the
  `APPLICATIONINSIGHTS_CONNECTION_STRING` environment variable. When set, the
  exporter uses it automatically — no additional configuration is needed in the
  collector pipeline.
- This is the **stable, documented path** recommended by both Microsoft and the
  OpenTelemetry Collector contrib maintainers.
- In Azure Container Apps, the connection string can be injected securely using
  a **Key Vault secret reference**. The ACA platform resolves the reference at
  container start, so the secret never appears in Bicep/Terraform definitions or
  container images.
- Rotation requires updating the Key Vault secret and restarting the container
  revision — a well-understood operational pattern.

### Managed Identity

- The `azuremonitor` exporter does **not** support Managed Identity natively.
  There is no built-in configuration option to authenticate via MI.
- MI-based authentication would require one of two workarounds:
  - **`azureauthextension`** — an OpenTelemetry Collector extension currently in
    **alpha** status in the contrib repository. It acquires Azure AD tokens and
    injects them into exporter requests. However, alpha components carry no
    stability guarantees, may introduce breaking changes between releases, and
    are not recommended for production workloads.
  - **Sidecar proxy** — a reverse proxy (e.g. Azure Identity Proxy) running
    alongside the collector that transparently attaches MI tokens to outbound
    requests. This adds operational complexity, an extra container, and a
    potential failure point.
- Neither approach is production-ready or widely adopted in the community.

## Decision

**Use the connection string via the `APPLICATIONINSIGHTS_CONNECTION_STRING`
environment variable, injected from a Key Vault secret reference in ACA.**

This is the only stable, production-grade authentication path for the
`azuremonitor` exporter today. It avoids alpha-quality dependencies and
unnecessary architectural complexity.

## Consequences

### Positive

- **Stable and documented** — uses the officially supported authentication
  mechanism with no experimental dependencies.
- **Simple configuration** — a single environment variable; no custom
  extensions or sidecar containers.
- **Secure secret handling** — the connection string is stored in Key Vault and
  injected at runtime via ACA secret references, keeping it out of source
  control and container images.
- **Proven operational model** — Key Vault secret rotation is a well-understood
  pattern across the organisation.

### Negative

- **Secret management overhead** — the connection string must be stored,
  rotated, and monitored in Key Vault, whereas MI would eliminate secrets
  entirely.
- **Not zero-trust** — connection strings are bearer credentials; a leaked
  string grants ingestion access until rotated.
- **Manual rotation** — secret rotation requires a Key Vault update and
  container revision restart (automatable but not automatic).

### Revisit triggers

- The `azureauthextension` reaches **GA** (stable) status in the OpenTelemetry
  Collector contrib repository.
- Microsoft adds native MI support to the `azuremonitor` exporter.
- Organisational policy mandates MI-only authentication for all Azure
  integrations.
