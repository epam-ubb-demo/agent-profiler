# T7.1.1 — ACA Environment Type: Consumption-only vs Workload Profiles

| Field     | Value                                      |
| --------- | ------------------------------------------ |
| **Status**  | Accepted                                 |
| **Date**    | 2025-01-10                               |
| **Feature** | F7.1 — Infrastructure Foundation         |
| **Task**    | T7.1.1 — ACA Environment Type Selection  |

## Context

The Centralized OpenTelemetry Gateway requires an Azure Container Apps (ACA)
environment to host the OTel Collector. ACA offers two environment types:

1. **Consumption-only** — serverless, pay-per-use pricing with no dedicated
   compute.
2. **Workload Profiles** — allows mixing a Consumption profile with dedicated
   compute profiles (D-series, E-series, etc.).

This spike investigates which environment type best suits our OTel Collector
workload, considering networking requirements, cost, and operational complexity.

## Investigation

### Consumption-only plan

- **Subnet sizing**: When deploying with a custom VNet, the Consumption-only
  environment requires a minimum `/23` subnet (512 addresses). This is a hard
  platform requirement.
- **Subnet delegation**: The subnet must **not** be delegated to
  `Microsoft.App/environments`. Delegation is only used by the Workload Profiles
  environment type; applying it to a Consumption-only environment causes
  deployment failures.
- **Additional CIDR parameters**: Three extra network parameters must be
  supplied at environment creation time:
  - `platformReservedCidr` — a `/22` or larger range for platform-internal
    services (e.g. `10.0.4.0/22`). Must not overlap with the infrastructure
    subnet.
  - `platformReservedDnsIP` — a single IP within the reserved CIDR for
    platform DNS (e.g. `10.0.4.2`).
  - `dockerBridgeCidr` — the Docker bridge network range
    (e.g. `172.17.0.1/16`). Must not overlap with any VNet address space.
- **Pricing model**: Pure pay-per-use — billed only for vCPU-seconds and
  GiB-seconds consumed. No baseline cost when idle.

### Workload Profiles + Consumption profile

- **Subnet sizing**: Requires only a `/27` subnet minimum (32 addresses) when
  using the Consumption profile within a Workload Profiles environment.
- **Subnet delegation**: The subnet **must** be delegated to
  `Microsoft.App/environments`.
- **Additional CIDR parameters**: Not required — the platform manages internal
  networking automatically.
- **Pricing model**: Adds a per-environment hourly charge on top of
  per-container usage costs. Dedicated profiles incur reserved compute costs
  regardless of utilisation.
- **Complexity**: Introduces profile management, scaling rules per profile, and
  additional Bicep/Terraform resources.

## Decision

**Use the Consumption-only environment type.**

The OTel Collector workload does not require dedicated compute, GPU access, or
guaranteed capacity. Consumption-only pricing aligns with our cost-optimisation
goals — we pay only for actual telemetry processing, with no idle baseline cost.

The larger subnet requirement (`/23`) and extra CIDR parameters are acceptable
trade-offs given the simpler operational model and lower cost.

## Consequences

### Positive

- **Lower cost** — no per-environment hourly charge; pure pay-per-use.
- **Simpler configuration** — no profile management or dedicated node scaling.
- **Sufficient for workload** — the OTel Collector is a lightweight, bursty
  workload well suited to serverless compute.

### Negative

- **Larger subnet** — the `/23` requirement consumes more VNet address space
  than the `/27` needed by Workload Profiles.
- **Extra CIDR parameters** — `platformReservedCidr`, `platformReservedDnsIP`,
  and `dockerBridgeCidr` must be explicitly managed and kept free of overlaps.
- **No subnet delegation** — the subnet must not be delegated, which differs
  from the Workload Profiles pattern and may cause confusion if engineers are
  accustomed to the latter.
- **No dedicated compute path** — if the workload later requires guaranteed
  CPU/memory or GPU, the environment must be recreated with Workload Profiles.

### Revisit triggers

- Sustained high-throughput telemetry requiring dedicated compute.
- Need for GPU-accelerated processing or specialised VM SKUs.
- Platform pricing changes that erode the cost advantage of Consumption-only.
