---
title: "ADR-0008: Sync Local Session Enrichment to Remote Workspace"
description: Decision record for syncing local-only Copilot CLI session enrichment to a remote Log Analytics workspace via OTLP through an OTel collector.
---

## Status

Accepted

## Date

2026-05-14

## Context

Remote Copilot CLI sessions can be reconstructed from OpenTelemetry spans, but several local-only fields are not available remotely: utilisation, compactions, `copilotVersion`, `reasoningEffort`, and tool results. As a result, remote viewers see an incomplete session record even when the underlying desktop app has the full local data.

We need a way to publish this enrichment data without overloading the existing telemetry model or weakening the privacy boundary around optional local artefacts.

## Decision

Push local-only session enrichment data to the Application Insights `AppTraces` table via OTLP/HTTP log records routed through the existing OTel collector.

### Why OTLP via the OTel collector

The existing OTel collector already handles authentication (Azure credentials), TLS encryption, and batching. By routing enrichment data through the collector via OTLP/HTTP (`POST /v1/logs`), we can:

- Deploy the desktop app without Azure SDK or credentials ﻗ°½ the app only needs the collector URL
- Land enrichment records in the `AppTraces` table (Application Insights), where they can be joined with existing session spans
- Distinguish enrichment records by setting `customDimensions["agent_profiler.enrichment"] == true`
- Benefit from the collector's existing export pipeline and infrastructure

### How sync is tracked

Each session stores a sidecar marker file, `.agent-profiler-sync.json`, alongside the local session data. The marker records:

- last synced row count (`lastSyncedRowCount`)
- last event ID
- categories already pushed

Marker updates are written atomically by creating a temporary file and renaming it into place. This prevents partial writes if the process exits mid-sync.

### v1 sync scope

v1 implements one-shot sync per session. Once a session has been synced, it is not synced again, even if new events are appended or the selected categories change later.

This is intentional for the initial release and keeps the implementation simple while the marker schema remains ready for a delta/resume model.

### Deferred delta sync

True delta/resume sync is deferred to a follow-up PR. The marker schema already carries the fields needed for that work, including `lastSyncedRowCount` and `categoriesPushed`, but v1 does not use them to resume or re-drive a previously synced session.

### Idempotency

Deduplication operates at three layers:

1. **Application-level**: The MarkerStore tracks `lastSyncedRowCount` to prevent re-syncing the same events.
2. **Wire-level**: Each enrichment record uses a deterministic `EventId` in the form `sessionId:category:index`, making retries safe and allowing the collector to deduplicate repeated uploads.
3. **Query-level**: A KQL safety net using `summarize arg_min(TimeGenerated, *) by EventId` deduplicates any records that slip through earlier layers.

### Triggers and settings

Sync can be started in two ways:

1. Manually via a **Sync now** action
2. Automatically when the desktop app's auto-sync setting is enabled

Users can choose which enrichment categories are uploaded:

- metadata ﻗ°½ enabled by default
- utilisation ﻗ°½ enabled by default
- compactions ﻗ°½ enabled by default
- tool results ﻗ°½ disabled by default for privacy

The `tool_result` category includes skill telemetry fields (`skillName`, `skillSource`, `skillContentLength`, `skillOutcome`, `skillErrorMessage`), merged in PR #359 and now available directly on the `ToolCall` type. These fields are conditionally included in the enrichment payload when present.

Authentication is handled by the OTel collector; the desktop app needs no Azure credentials.

## Alternatives considered

### Azure Logs Ingestion API (original v1 approach)

Originally implemented but replaced in favour of OTLP transport. The Logs Ingestion API required three configuration fields (Data Collection Endpoint, Data Collection Rule Immutable ID, DCR Stream Name) and the `@azure/monitor-ingestion` SDK with `DefaultAzureCredential` on the desktop. This was rejected because:

- Complex provisioning burden for users
- Required Azure credentials on every desktop machine
- The OTel collector already handles the Azure authentication and export pipeline

### Extend OpenTelemetry spans

Rejected. This would pollute existing telemetry with local-only enrichment and create a tighter coupling between the remote session model and the desktop app's internal data structures.

### Use Application Insights custom events directly

Rejected. Custom events do not provide the same bulk ingestion workflow and are a poorer fit for the volume and shape of session enrichment data.

### Store enrichment in Azure Blob Storage

Rejected. Blob storage would not provide a native KQL join path with the session data, making remote analysis and viewer correlation much harder.

## Consequences

- Remote viewers can see a complete session record, including local-only enrichment.
- The desktop app uses Node.js built-in `fetch` to send OTLP JSON ﻗ°½ no external SDK dependency.
- The desktop app requires only the OTel Gateway URL to be configured.
- The desktop settings UI gains sync controls, OTel Gateway URL configuration, and status feedback.
- The system introduces a small amount of per-session state management through the marker file, but v1 sync is one-shot per session.
- Delta/resume sync remains a known limitation and follow-up item.
